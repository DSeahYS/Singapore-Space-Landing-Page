Singapore’s NEqO Satellite Constellation
Core satellites to feature
ST Engineering Geo‑Insights is the exclusive provider of imagery and analytics for Singapore’s Near‑Equatorial Orbit (NEqO) constellation, which comprises TeLEOS‑1, TeLEOS‑2, DS‑EO, NeuSAR, and DS‑SAR (often styled DS‑SAR). This multi‑modal constellation offers both high‑resolution optical imagery (down to 50 cm) and SAR imagery (1–2 m depending on satellite and mode), with high revisit rates over the equatorial region that includes Southeast Asia and key maritime routes.

Key technical facts you can surface in the UI:

TeLEOS‑1 – Singapore’s first commercial Earth‑observation satellite in NEqO, carrying an electro‑optical camera with about 1 m ground sampling distance for maritime monitoring, homeland security, and disaster management.

TeLEOS‑2 – A PolSAR (polarimetric SAR) satellite developed with DSTA, featuring a made‑in‑Singapore SAR payload capable of 1 m resolution, full polarimetry, and day‑night all‑weather imaging with around 14 passes per day at 10° inclination.

DS‑EO – An electro‑optical multi‑spectral satellite acquired jointly by ST Engineering and DSTA from Satrec Initiative, providing 0.5 m panchromatic and 2 m multispectral imagery with up to six daylight revisits per day in NEqO at about 570 km altitude.

NeuSAR – Singapore’s first fully polarimetric SAR microsatellite, designed by DSO with OSTIn support; provides 1.5 × 0.5 m spot‑mode and 2 × 2 m strip‑mode SAR imaging and demonstrates high‑performance SAR on a 160 kg bus.

DS‑SAR (DS‑SAR/DS‑SAR) – A SAR Earth‑observation satellite built by Israel Aerospace Industries for Singapore’s DSTA and ST Electronics, launched in July 2023 into near‑equatorial LEO around 530 km with 5° inclination, supporting maritime security and oil‑spill detection while enhancing ST’s commercial imagery services.

These satellites are operated for both national and commercial purposes, with ST Engineering, DSTA, DSO, and OSTIn all playing roles in development, operations, or commercialisation.

Glassmorphism and skeuomorphism in CSS
Glassmorphism ("frosted glass" UI) is commonly implemented using semi‑transparent backgrounds, gradients, and backdrop-filter: blur(...) in combination with subtle borders and highlights, as explained in tutorials and guides. A typical pattern uses a dark background image or gradient, then overlays panels with rgba or gradient backgrounds, a 1–2 px border in a light colour (for you: Mist Gray), and blur to create the glass effect.

Skeuomorphic controls like knobs and switches can be recreated using layered box‑shadows (outer and inset), gradients, and transform animations so that buttons visually depress, combined with transitions for frictionless motion. By nesting skeuomorphic elements inside glassmorphic panels, your design can achieve the "Tactile Glass" hybrid: Space Black body, Liquid Glass panels, and physical‑feeling controls atop them.

## Common GitHub pattern

Most public satellite trackers follow one of three patterns:

- **Pattern 1: Pure front end** — fetch TLE/GP data from CelesTrak, propagate with `satellite.js`, render in CesiumJS, and update positions on a timer. [github](https://github.com/boostedspaceprogram/sat-vis)
- **Pattern 2: Front end + preprocessor** — fetch from Space-Track/CelesTrak server-side, transform to JSON or CZML, then serve cleaner payloads to the front end. [github](https://github.com/gamergenic/ASATDebrisCsExample)
- **Pattern 3: React wrapper** — use Resium or Cesium inside React, keep satellites in state, and separate propagation logic into services/hooks. [github](https://github.com/jester22/satellite-tracker)

For your page, Pattern 2 is the safest long-term architecture because it avoids CORS surprises, protects credentials, and lets you normalize satellite metadata for TeLEOS/DS-EO/NeuSAR-specific storytelling. [github](https://github.com/cassova/satellite-czml)

What to use per feature
For the Sovereign Orbit Tracker, the cleanest production path is: backend fetches CelesTrak or Space-Track, stores normalized orbital records, then front end uses satellite.js to propagate and CesiumJS to render real-time positions and orbit lines.

For the SAR & Optical Explorer, use Sentinel Hub for real public imagery search/render demos and present TeLEOS, NeuSAR, DS-SAR, and ICEYE as metadata-driven “mission feeds” unless you have actual licensed imagery access.

For the SSA Monitor, use simulated conjunction logic over public orbital feeds first, because LeoLabs and Digantara appear better suited as partnership-backed enterprise integrations than open developer APIs.

Request patterns

1. CelesTrak
CelesTrak’s GP queries use a simple URL pattern: gp.php?{QUERY}=VALUE&FORMAT={FORMAT} with JSON, CSV, XML and other formats supported.

Example:

text
GET <https://celestrak.org/NORAD/elements/gp.php?CATNR=25544&FORMAT=JSON>
Other useful forms:

text
GET <https://celestrak.org/NORAD/elements/gp.php?GROUP=STATIONS&FORMAT=JSON>
GET <https://celestrak.org/NORAD/elements/gp.php?NAME=COSMOS> 2251 DEB&FORMAT=JSON
GET <https://celestrak.org/NORAD/elements/gp.php?INTDES=2020-025&FORMAT=JSON-PRETTY>
Use this as your easiest live demo source because it requires no auth and is already used widely in open satellite trackers.

1. Space-Track
Space-Track is better handled server-side because it requires authentication and is built for more serious catalog and orbit workflows than a simple front-end fetch.

Use cases:

scheduled TLE refresh jobs,

object metadata enrichment,

future conjunction-related workflows,

fallback when CelesTrak grouping is insufficient.

1. Sentinel Hub
Sentinel Hub exposes REST APIs for imagery archives, rendered images, statistical analysis and batch processing, and its Process API requires account credentials via OAuth client setup.

Use it for:

imagery scene search by area and date,

rendered preview panels,

public sandbox code samples,

“compare cloudy optical vs SAR” interaction demos.

1. NASA CMR-STAC
NASA’s CMR-STAC is strong for public developer education because it is STAC-based and easy to position as “public Earth data discovery.”

Use it for:

sample search cards,

code examples in your sandbox,

alternate open-data mode when commercial feeds are unavailable.

GitHub implementation patterns
Open-source repos tend to split responsibilities in a repeatable way:

Fetcher layer pulls TLE data on a schedule or at app load.

Propagation layer uses satellite.js to convert orbital elements into time-based positions.

Visualization layer renders in Cesium or Resium and updates entities over time.

Preprocessing layer sometimes converts TLE into CZML for smoother time playback inside Cesium.

That means your codebase should likely have:

services/orbits.ts

services/imagery.ts

services/ssa.ts

lib/propagation.ts

adapters/celestrak.ts

adapters/sentinelHub.ts

adapters/nasaStac.ts

adapters/partnerPlaceholder.ts

Recommended production architecture
The most developer-safe architecture is:

Backend proxy/API layer

Fetch CelesTrak and Space-Track data server-side.

Cache normalized orbital data.

Hide all secrets for Sentinel Hub and any future partner systems.

Front-end app

CesiumJS for the globe and scene transitions.

satellite.js for propagation and orbit animation patterns validated by GitHub examples.

React + Resium only if you want component abstraction; otherwise plain Cesium is simpler for performance-heavy hero scenes.

Data modes

publicMode: CelesTrak + NASA + Copernicus only.

partnerMode: adds ICEYE, LeoLabs, Digantara, Space-Track enrichment.

Exact recommendation
If you want the fastest credible build, use this stack first:

Tracker: CelesTrak + satellite.js + CesiumJS.

Imagery explorer: Sentinel Hub Catalog + Process API.

Public dataset sandbox: NASA CMR-STAC + Copernicus API docs.

SSA demo: simulated conjunction engine using public orbit feeds, with LeoLabs/Digantara shown as partnership modules, not live dependencies yet.
